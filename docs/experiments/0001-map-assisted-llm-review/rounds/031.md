# Round 31

- Subject: PR #147 — new **Markers** section in the `?` help overlay
  listing every left-tree marker/badge (`chg:`, `api:`, `fan-in:`, `!`,
  `N tests`, `(cycle)`, `warn:`, `split:`, `~`, kind abbreviations, and
  the `[test]`/`(skipped: …)` badges) with its real ratatui style, so
  the reference reads as identical to what the tree renders — no more
  glossary/tree drift. Implementation extracts style-producer helpers
  (`test_badge_style`, and the pre-existing
  `cyan_badge_style`/`warning_badge_style`/`split_badge_style` pattern)
  and wires them into a single `marker_swatch_spans` match, so the two
  sites (legend swatch and tree row) share one source of truth. The
  overlapping Glossary entries the Markers section now covers were
  dropped in the same commit.
- Protocol note: same cost-reduced configuration as rounds 27-30 — one
  static reviewer combining the map-assisted pass and the independent
  pass, one separate dynamic-verification agent — but this PR needed
  **two rounds** of the full static + dynamic pipeline because a second
  silent fallthrough shipped past the first review.
- **Map delivery**: self-serve, variant (b) — the static reviewer built
  rinkaku from a trusted `main` checkout itself and ran that binary's
  `rinkaku --base main` inside the branch worktree before reading any
  code.
- **Round 1, static pass**: found that the `(cycle)` swatch in the
  legend fell through `marker_swatch_spans`' default `Span::raw` arm
  and rendered as plain text, while directory rows in the actual tree
  render `(cycle)` as bold yellow. **BLOCKER** — the feature's whole
  point is that the legend swatch matches what the tree renders, and
  this one didn't. Fixed in `3bc8d22` by adding the explicit `(cycle)`
  arm, adding per-swatch style tests for the four previously-unpinned
  entries (`(cycle)`, `warn:`, `split:`, `~`), and dropping the
  Glossary duplicates.
- **Round 1, dynamic pass**: captured every swatch via
  `tmux capture-pane -e` and confirmed byte-for-byte SGR match against
  the corresponding tree-row style for the entries covered at that
  point.
- **Round 2, static pass** (after the round-1 fix landed): a fresh
  static reviewer independently enumerated every entry in
  `MARKER_LEGEND` and re-checked each against `marker_swatch_spans`,
  and found a **second silent fallthrough** — the `[test] (N symbols)`
  entry also hit the default `Span::raw` arm and rendered unstyled in
  the overlay while the real tree row renders it Magenta via
  `test_badge_span`. **BLOCKER.** Fixed in `90891eb` by extracting
  `test_badge_style` from `test_badge_span` (mirroring the existing
  `cyan_badge_style`/`warning_badge_style`/`split_badge_style` pattern)
  and adding the explicit legend arm; the two call sites now share one
  style-producer function.
- **Round 2, dynamic verification** (final): re-captured all 16
  swatches after the second fix via `tmux capture-pane -e` and
  confirmed byte-for-byte SGR match with the corresponding tree rows,
  including the newly-fixed `[test]` entry. Clean.
- Severity summary: 2 blockers across two rounds (both silent
  swatch-fallthrough defects of the same shape, both fixed same PR),
  0 should-fix, 0 nits.
- Cost: implementation ~209k tokens (initial) + ~116k (round-1 fix-up)
  + ~96k (round-2 blocker fix); static reviewers ~102k tokens
  (round 1) + ~107k tokens (round 2); dynamic reviewer ~158k tokens
  (combined round 1 + round 2). Not a harness-timed comparison;
  wall-clock not measured.
- Takeaway: the whole point of this feature is drift prevention
  between the tree and its documentation, and **the drift-prevention
  property was itself not enforced by test** — it was enforced by the
  author remembering to add a matching arm to `marker_swatch_spans`
  for every entry added to `MARKER_LEGEND`. Because that match's
  default arm is `Span::raw`, the failure mode of forgetting an arm is
  silent: no test breaks, no error surfaces, the legend just renders
  the missing entry as plain text. Two separate silent fallthroughs
  (`(cycle)` in round 1, `[test]` in round 2) shipped past their
  respective implementation-time reviews under exactly this pattern,
  each caught only because a static reviewer re-enumerated legend
  entries against style code by eye. This generalises the round-17/29
  lesson (side-channel drift is a recurring failure mode) to a related
  class: **a "match this other site" invariant that lives in a
  default-fallthrough match arm is drift-hostile by construction, and
  writing per-entry tests one at a time does not close the gap because
  the next entry to be added is exactly the one that will lack a
  test.** A follow-up worth recording for a future PR: either an
  exhaustive test that iterates every `MARKER_LEGEND` swatch and
  asserts its produced style is non-default (any `Span::raw`
  fallthrough fails the whole suite), or a construction-time design
  where the style is looked up by swatch key (e.g. `HashMap<&str,
  Style>` populated at build time, or a `#[derive]`-enforced total
  match) so that a missing entry fails at construction rather than at
  render.
