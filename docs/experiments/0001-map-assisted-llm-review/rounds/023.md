# Round 23

- Subject: this PR's own diff — `--format mermaid` grows a `removed`
  node class for deleted symbols (ADR 0035), and a new `--format
  digest` output (ADR 0036) replaces the full Markdown `<details>` in
  `action.yaml`/`compose_and_post_comment.sh`'s sticky PR comment. Same
  shape as round 10 (this PR's own diff, `--format mermaid` + the
  GitHub Action's introducing PR): a mix of `rinkaku-core` render-layer
  Rust, CLI plumbing, and non-Rust orchestration surface (`action.yaml`,
  `compose_and_post_comment.sh`, ADRs, docs).
- Protocol note: three-angle review per this repository's `CLAUDE.md`
  ("Reviewing changes"), not a harness-timed A/B pair — same as round
  17. Blocker count: 0. Two should-fix findings, both addressed in this
  round's follow-up commits.
- **Map-assisted pass**: `rinkaku --base main` (trusted build, not the
  branch under review) reported the new `render_digest`/`removed`-node
  code as high-fan-in, correctly clustering `rinkaku-core/src/render/
  mermaid.rs` and the new `digest.rs` as the diff's core. No `## File
  size warnings` entry (mermaid.rs sits at 1025 lines — see below —
  comfortably under the 1500 watch threshold). The map's contribution
  this round was narrower than round 10's: it correctly flagged the
  render-layer surface as the place to read deeply, but the two
  should-fix findings both lived in *presentation glue* the map has no
  vocabulary for — a duplicated `<summary>`/`### API changes` heading
  in a shell heredoc, and dead env-var wiring in a composite action's
  YAML. Neither is a Rust symbol change the map's signature/fan-in
  model can represent. One genuine nit surfaced from reading the
  composed-comment shape the map's mermaid/digest output pointed at:
  the summary-duplication finding (nit 3 below) was found by treating
  the map's own rendered digest sample as "what will actually appear
  under this summary line" and noticing the heading repeated.
- **Independent pass**: plain review of the diff (ADRs 0035/0036,
  `mermaid.rs`, `digest.rs`, `cli.rs`, `action.yaml`,
  `compose_and_post_comment.sh`) against repo conventions (assert
  style, no library-code `unwrap`/`expect`, ADR cross-linking). Found
  the two should-fix items dynamic/map passes didn't:
  1. `digest.rs`'s `added_line`/`signature_changed_line`/`removed_line`
     rendered only a bare symbol name — two files defining a same-named
     symbol (`new`, `run`, a common test helper) would produce
     indistinguishable digest entries, inconsistent with
     `render_markdown`'s own `tree_label`/`removed_symbol_label`, which
     already append `(path)` for exactly this reason.
  2. `action.yaml` still wired `MARKDOWN_PATH` into the "Compose and
     post sticky PR comment" step's env block, but
     `compose_and_post_comment.sh` had already stopped reading that
     variable once the `<details>` body switched to the digest — dead
     wiring, and inconsistent with ADR 0036's own "`MARKDOWN_PATH` is
     replaced by `DIGEST_PATH`" text.
  This pass is where both should-fix findings came from — consistent
  with this experiment's standing pattern (rounds 1, 3, 9) that the
  independent pass is strongest on line-level/consistency defects a
  map's symbol-level view doesn't encode.
- **Dynamic verification**: built the branch's binary and exercised
  every fallback branch of `compose_and_post_comment.sh` via
  `DRY_RUN=1` against real `--format mermaid`/`--format digest` output
  generated from this branch's own diff — normal case, empty digest (no
  contract changes), digest path unset (bootstrap fallback), oversized
  mermaid (`>32768` bytes), and — the finding this pass specifically
  produced that neither static pass had flagged — **a path variable set
  to a nonexistent file**. Reproduced the crash first (`cat` under
  `set -euo pipefail` aborting the whole script with no fallback text
  at all, confirmed via a standalone repro before the fix), then
  verified the fix (`-f` existence guards) degrades gracefully instead.
  Separately, dynamic verification of the ADR 0035 claim itself found
  the sharpest finding of the round: built two throwaway git repos to
  compare **partial file deletion** (file survives, one symbol removed
  — `report.removed` populated correctly, mermaid renders a `removed`
  node) against **whole-file deletion** (`git rm` the whole file —
  `report.skipped` gets `{"reason":"deleted"}`, `report.removed` stays
  `[]`, and `--format mermaid` renders `%% no symbols` for that file).
  ADR 0035's Decision text claimed the file-level fallback "represents
  removed-only files" without that distinction, which is true only for
  the survives-but-emptied case — not for outright deletion. This is a
  pre-existing gap traced to `pipeline.rs`'s `analyze_diff`, which
  `break`s out of a `ChangeKind::Deleted` file (recording it only in
  `report.skipped`) *before* `classify_against_base` (ADR 0014) ever
  runs, so no symbol from a wholly-deleted file ever reaches
  `report.removed` at all — inherited unchanged by ADR 0036's
  `--format digest`, which reads the same field.
- **Fix applied this round**: (1) `digest.rs` gained a `located_name`
  helper (`"{name} ({path})"`) threaded through all three line-builder
  functions, plus a new test pinning the two-files-same-name case; ADR
  0036's sample output updated to match. (2) `action.yaml`'s dead
  `MARKDOWN_PATH` env entry removed; the `<details><summary>` text
  changed from "API changes" (duplicating `render_digest`'s own
  `### API changes` heading one line below it) to "Details". (3) ADR
  0035's Decision and Consequences sections rewritten to state the
  whole-file-deletion limitation explicitly, cross-linked to a new
  follow-up issue
  ([hiro-o918/rinkaku#115](https://github.com/hiro-o918/rinkaku/issues/115));
  ADR 0036's Consequences also notes it inherits the same gap. (4)
  `compose_and_post_comment.sh` gained `-f` existence guards on both
  `MERMAID_PATH`/`DIGEST_PATH` (verified the crash it fixes reproduces
  without the guard) and a softened fallback message ("older rinkaku
  binary or generation failed" instead of asserting the binary
  "predates `--format digest`," which was wrong for the
  probe-succeeded-but-invocation-failed case). `make test`/`make lint`/
  `shellcheck`/`actionlint` all green after every fix.
- Severity summary: 0 blockers. 2 should-fix (digest path
  disambiguation, dead `MARKDOWN_PATH` wiring), both fixed this round.
  2 nits (summary/heading duplication, softened fallback wording), both
  fixed alongside a bonus robustness fix (missing-path crash guard)
  that dynamic verification surfaced but no static pass had asked for.
- Takeaway: the three angles split along the same lines round 10 (this
  experiment's other "own-diff" subject) already established, but with
  the roles slightly reassigned. The map correctly routed attention to
  the render-layer core and stayed silent on the shell/YAML surface —
  consistent with rounds 10/17's standing point that the map's silence
  on non-signature surfaces is a coverage boundary, not a defect,
  and must not be read as "nothing to find there." The independent
  pass caught both should-fix findings (a missing disambiguation
  convention, a dead wiring/doc mismatch) that require noticing
  something *absent* — a path suffix that should have been there, an
  env var that used to matter and no longer does — which is exactly
  the class of defect a signature/fan-in map has no way to represent
  either way. Dynamic verification did what it has done every round
  since round 5: it did not just confirm a suspected defect, it *found*
  two the static passes missed entirely (the set-but-missing-path
  crash; the ADR's own claim being falsifiable by simply constructing
  the counter-example repo) — reinforcing that an ADR's Decision prose
  is itself a review target whose claims should be verified against
  running code, not accepted as accurate documentation of the feature
  it introduces.
