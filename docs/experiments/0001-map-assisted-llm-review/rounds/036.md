# Round 36

- Subject: PR #164 — split (side-by-side) view for the source screen's
  diff overlay (ADR 0049): reconstructs a file's old-side text by pure
  reverse-application of its already-parsed hunks onto the new-side
  full text (no new IO), pairs it against the new side via the diff
  pane's own `pair_hunk_lines`, and wires `v`/`V` to toggle it from
  `Screen::Source` too.
- Protocol: medium-tier change per CLAUDE.md's process-weight rule (a
  new rendering mode extending an existing pattern) — one map-assisted
  static-review pass on top of the small-change TDD flow, no separate
  independent pass or dynamic-verification agent. Dynamic verification
  (non-TTY stdin/stdout, empty diff, conflicting flags, missing base
  ref, non-TTY `--tui`) was carried out during the implementation phase
  itself, per this repository's own "someone must run it, either pass
  may carry this step" rule.
- **Map delivery**: `rinkaku --base origin/main`, built from a trusted
  `main` checkout, run inside the branch worktree before reading the
  diff.
- **What the map surfaced**: 55 changed symbols across 12 files, no new
  `## File sizes` warn/split entries (the two pre-existing `watch`/
  `warn` entries, `handle_key.rs` and `lib.rs`, predate this PR and
  grew by low double-digit and single-digit lines respectively). High
  fan-in concentrated on the three new types in `source_split.rs`
  (`SourceSplitRow`, `SourceSplitLine`, `SourceSplitRowKind`), which
  routed the reviewer into that module's reverse-application and
  pairing logic first.
- **Blocker, found by line-level reading, not the map**: in
  `source_screen.rs`'s `source_split_lines`, the gate deciding which
  cells get token highlighting was keyed on `SourceSplitRowKind::Unchanged`
  rather than on which side a cell is — so a `Changed` row's new-side
  (`Added`) cell rendered as plain unhighlighted text even though its
  `line_number` is a valid `token_highlights` index, the same index the
  unified overlay's own `Added` row already looks up. This is exactly
  the line a reviewer opens split view to see. **Nothing about this
  defect shows up on the signature surface** — the function's own
  signature was unchanged, and the map's fan-in ranking had already
  done its job by routing attention into `source_split.rs`'s row-kind
  enum; finding the defect itself required reading `source_split_lines`
  side by side with the unified overlay's already-correct `Added`-row
  handling and noticing the two disagreed on when highlighting applies.
  A fresh instance of "the map allocates attention; it is not a
  verifier."
- **Same-finding nit**: the doc comment on `source_split_side_line`
  asserted highlighting was valid "only for an `Unchanged` row's
  new-side cell" — a factually wrong justification for the very gate
  that caused the regression, the kind of confidently-wrong comment
  that makes a defect harder to spot on a first read rather than easier.
- **Manually traced, no issues found**: `reconstruct_old_lines`'s
  reverse-application walk (each hunk's `Context`/`Removed` lines
  replayed as old-side text, `Added` dropped, drift detected the same
  way the unified overlay already does), the split-mode scroll/clamp
  path through `render_scrollable_pane`, and the narrow-pane/drift
  fallback to unified rendering.
- Severity summary: 1 blocker (highlighting gate keyed on the wrong
  condition), 1 same-finding nit (the comment justifying that gate).
  Both fixed in `0d444b7`, TDD (a failing render-level test confirmed
  Red against the pre-fix code, asserting the new-side keyword's
  foreground color, then Green after the fix). `make test`: 739 passed
  (up from 738). `make lint`: clean.
- Takeaway: this round pairs with round 35 as a second instance of the
  same shape — high fan-in on a diff's new core types correctly routed
  the reviewer into the right module, but the actual defect (two code
  paths disagreeing about when a shared piece of state applies) had no
  footprint on any signature, contract marker, or fan-in count, and
  surfaced only from comparing the new split-rendering path against the
  unified path's already-correct behavior line by line. The
  accompanying wrong doc comment is also consistent with round 34's
  "an inaccurate comment describing the defective code as safe" pattern
  — the comment did not cause the bug, but it actively worked against
  a reader noticing it.
